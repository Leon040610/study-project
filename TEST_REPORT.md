### 4.1 阻断级问题 (Blocker)

#### ISSUE-001 | 前后端数据模型不一致导致 CRUD 失败

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-001 |
| **类型** | Bug / 数据一致性 |
| **严重程度** | 阻断 |
| **影响范围** | 全部 CRUD 操作（目标、计划、任务的创建/更新） |
| **文件位置** | `vue-frontend/src/stores/data.ts` 全文; `api/routes/goals.ts`, `api/routes/plans.ts`, `api/routes/tasks.ts` |

**描述:**  
前端 TypeScript 接口使用 camelCase 字段名（如 `targetDate`, `startDate`, `completedDate`），后端接口使用 snake_case 字段名（如 `target_date`, `start_date`, `completed_date`）。前端 `addGoal()`、`addPlan()`、`addTask()` 直接将前端对象发送到后端，后端无法正确解析字段。

**具体不一致:**

| 实体 | 前端字段 | 后端字段 |
|------|----------|----------|
| Goal | `targetDate`, `completedDate`, `abandonedDate` | `target_date`, `completed_date`, `abandoned_date` |
| Plan | `category`, `goalTitle`, `startDate`, `endDate` | `goal`, `start_date`, `end_date`, `daily_minutes`, `progress` |
| Task | `planTitle`, `date`, `completedDates` | `plan_id`, `due_date`, `priority` |

**复现步骤:**
1. 登录系统
2. 创建一个新目标，填写标题、分类、目标日期
3. 检查后端存储的 goals.json 文件

**实际结果:** 后端接收到的字段名为 `targetDate`，但后端期望 `targetDate`（goals.ts 第 30 行 `const { title, description, category, targetDate } = req.body`），虽然此处恰好匹配，但 Plan 和 Task 的字段完全不匹配。

**预期结果:** 前后端数据模型字段名一致，或通过 DTO 层进行转换。

**修复建议:** 统一前后端数据模型字段命名规范，建议全部使用 camelCase，并在后端添加 DTO 转换层。

#### ISSUE-002 | 后端缺少任务创建接口 (POST /tasks)

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-002 |
| **类型** | Bug / 功能缺失 |
| **严重程度** | 阻断 |
| **影响范围** | 计划页面创建计划时自定义任务的创建 |
| **文件位置** | `api/routes/tasks.ts` (无 POST 路由) |

**描述:**  
前端 `data.ts` 的 `addTask()` 方法调用 `api.post('/tasks', newTask)`，但后端 `tasks.ts` 路由文件中没有定义 `POST /` 路由。导致前端创建计划时添加的自定义任务无法保存到后端。

**复现步骤:**
1. 进入"学习计划管理"页面
2. 点击"创建计划"
3. 填写计划信息和自定义任务
4. 点击"确定"提交

**实际结果:** 计划创建成功，但自定义任务创建失败（API 返回 404），任务仅存在于前端内存中，刷新后丢失。

**预期结果:** 后端提供 `POST /tasks` 接口，正确创建并持久化任务。

**修复建议:** 在 `api/routes/tasks.ts` 中添加 `POST /` 路由，接收任务数据并持久化。

#### ISSUE-003 | 计划编辑功能未实现 - 编辑模式始终创建新计划

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-003 |
| **类型** | Bug / 功能逻辑错误 |
| **严重程度** | 阻断 |
| **影响范围** | 计划管理页面的编辑功能 |
| **文件位置** | `vue-frontend/src/views/Plans.vue` 第 233-282 行 `handleSubmit()` |

**描述:**  
`editPlan()` 方法将计划数据加载到表单并设置 `editingPlan`，但 `handleSubmit()` 方法中没有检查 `editingPlan` 是否存在，始终执行 `dataStore.addPlan()` 创建新计划，而非更新现有计划。

**复现步骤:**
1. 在计划列表中点击某个计划的"编辑"按钮
2. 修改计划名称
3. 点击"确定"提交

**实际结果:** 创建了一个新的计划，原计划未被修改。

**预期结果:** 更新现有计划，不创建新计划。

**修复建议:** 在 `handleSubmit()` 中检查 `editingPlan.value`，若存在则调用 `dataStore.updatePlan()`，否则调用 `dataStore.addPlan()`。

#### ISSUE-004 | 验证码接口返回明文验证码

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-004 |
| **类型** | 安全漏洞 |
| **严重程度** | 严重 |
| **影响范围** | 所有用户 |
| **文件位置** | `api/routes/auth.ts` 第 155-170 行 |

**描述:**  
`GET /api/auth/captcha` 接口在响应中同时返回 `image`（SVG 图片）和 `captcha`（明文验证码文本）。验证码的目的就是防止自动化攻击，明文返回使验证码完全失效。

**复现步骤:**
1. 调用 `GET /api/auth/captcha`
2. 查看响应 JSON

**实际结果:** 响应包含 `{ image: "...", captcha: "ABCD" }`，验证码明文可见。

**预期结果:** 验证码应存储在服务端 session 或 JWT 中，响应只返回图片，不返回明文。

**修复建议:** 移除响应中的 `captcha` 字段，将验证码存储在服务端（session 或 Redis），验证时从服务端读取。

#### ISSUE-005 | JWT 密钥使用硬编码默认值

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-005 |
| **类型** | 安全漏洞 |
| **严重程度** | 严重 |
| **影响范围** | 所有认证相关功能 |
| **文件位置** | `api/routes/auth.ts` 第 52, 66 行; `api/middleware/auth.ts` 第 22 行 |

**描述:**  
JWT 签名密钥使用 `process.env.JWT_SECRET || 'secret'`，当环境变量未设置时使用硬编码的 `'secret'`。攻击者可以利用已知的默认密钥伪造任意 JWT token，获取管理员权限。

**实际结果:** 若未配置 `JWT_SECRET` 环境变量，使用 `'secret'` 作为密钥，可被轻易破解。

**预期结果:** 启动时强制要求 `JWT_SECRET` 环境变量，未设置时拒绝启动。

**修复建议:** 在应用启动时检查 `JWT_SECRET` 是否存在，不存在则抛出错误终止启动。

#### ISSUE-006 | 用户 ID 生成使用 Math.random() 非密码学安全

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-006 |
| **类型** | 安全漏洞 |
| **严重程度** | 严重 |
| **影响范围** | 所有用户、目标、计划、任务 |
| **文件位置** | `api/routes/auth.ts` 第 44 行; `api/routes/goals.ts` 第 28 行; `api/routes/plans.ts` 第 17 行 |

**描述:**  
所有实体的 ID 使用 `Math.random().toString(36).substring(2, 15)` 生成。`Math.random()` 不是密码学安全的随机数生成器，生成的 ID 短（约 13 字符）且可预测，存在碰撞风险和 ID 枚举攻击风险。

**修复建议:** 使用 `crypto.randomUUID()`（Node.js 内置）生成 UUID v4，确保唯一性和不可预测性。

#### ISSUE-007 | 无 API 速率限制 - 暴力破解风险

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-007 |
| **类型** | 安全漏洞 |
| **严重程度** | 严重 |
| **影响范围** | 登录、注册接口 |
| **文件位置** | `api/routes/auth.ts` 全文 |

**描述:**  
登录和注册接口没有速率限制，攻击者可以无限次尝试密码，存在暴力破解风险。

**修复建议:** 添加 `express-rate-limit` 中间件，限制登录接口每 IP 每分钟最多 5 次请求。

#### ISSUE-008 | 后端无输入验证和消毒 - XSS/注入风险

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-008 |
| **类型** | 安全漏洞 |
| **严重程度** | 严重 |
| **影响范围** | 所有 POST/PUT 接口 |
| **文件位置** | `api/routes/auth.ts`, `api/routes/goals.ts`, `api/routes/plans.ts` |

**描述:**  
后端接口直接使用 `req.body` 中的值，没有使用验证库（如 Joi、Zod）进行输入验证和消毒。用户可以在标题、描述等字段中注入恶意脚本，导致存储型 XSS。

**修复建议:** 引入 Zod 或 express-validator 进行输入验证，对所有字符串字段进行长度限制和 HTML 转义。

#### ISSUE-009 | FileStore 并发写入时静默丢弃数据

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-009 |
| **类型** | Bug / 数据丢失 |
| **严重程度** | 严重 |
| **影响范围** | 所有数据持久化操作 |
| **文件位置** | `api/utils/fileStore.ts` 第 57-59 行 |

**描述:**  
`save()` 方法使用简单的 `lock` 布尔值防止并发写入。当 `lock` 为 `true` 时，`save()` 调用被静默跳过（仅打印 warning），导致数据丢失。在并发请求场景下，多个写入操作可能被丢弃。

**实际结果:** 并发写入时部分数据不会被保存到文件。

**预期结果:** 并发写入应排队执行，不丢弃任何写入操作。

**修复建议:** 使用写入队列（Promise 链）替代布尔锁，确保每次写入都被执行。

#### ISSUE-010 | 前端 API 拦截器 401 处理使用 window.location 导致 SPA 状态丢失

|------|-----|
| **ID** | ISSUE-010 |

#### ISSUE-012 | 计划删除时未删除关联任务（后端）

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-012 |

| **严重程度** | 一般 |
| **影响范围** | 计划删除功能 |
| **文件位置** | `vue-frontend/src/views/Plans.vue` 第 297-301 行 `deletePlan()` |

**描述:**  
前端 `deletePlan()` 调用 `dataStore.deletePlan()` 删除计划，然后手动从前端 tasks 数组中过滤掉关联任务。但后端 `DELETE /plans/:id` 虽然会删除关联任务（plans.ts 第 104-107 行），前端 `dataStore.deletePlan()` 只调用了 `api.delete('/plans/${id}')`，没有同步删除本地 tasks。Plans.vue 中的手动删除是临时补丁，但如果用户在其他页面操作任务，可能引用已删除的计划。

**修复建议:** 添加 `loading` 状态变量和 `v-loading` 指令，添加空数据时的占位提示。

#### ISSUE-018 | 前端乐观更新无回滚机制

| 字段 | 值 |
|------|-----|
| **ID** | ISSUE-018 |
| **类型** | Bug / 数据一致性 |
| **严重程度** | 一般 |
| **影响范围** | 所有增删改操作 |
| **文件位置** | `vue-frontend/src/stores/data.ts` |

**描述:**  
`addGoal()`、`addPlan()`、`addTask()` 等方法先更新本地状态再调用 API。如果 API 调用失败（catch 块为空），本地状态已改变但服务器未更新，导致数据不一致。没有回滚机制恢复原始状态。

**修复建议:** 在 API 调用失败时回滚本地状态到操作前的值。
